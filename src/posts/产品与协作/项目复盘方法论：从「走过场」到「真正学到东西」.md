# 项目复盘方法论：从「走过场」到「真正学到东西」

复盘是团队成长最有效的手段，但大多数复盘会变成了「成绩汇报会」或「甩锅大会」。本文从复盘框架到会议组织，帮你把复盘从走过场变成真正的学习工具。

---

## 一、复盘四步法

### 1.1 GRAI 复盘框架

```
GRAI 复盘法：

  G (Goal)     回顾目标：当初要做什么？
  R (Result)   评估结果：实际做成了什么？
  A (Analysis) 分析原因：为什么是这个结果？
  I (Insight)  总结规律：下次怎么做更好？

  ┌─────────────────────────────────────┐
  │  Step 1: 回顾目标                     │
  │  - 项目目标是什么？                    │
  │  - 成功标准是什么？                    │
  │  - 时间/成本/质量约束是什么？           │
  ├─────────────────────────────────────┤
  │  Step 2: 评估结果                     │
  │  - 实际达成了什么？                    │
  │  - 与目标差距多少？                    │
  │  - 哪些超预期？哪些不如预期？           │
  ├─────────────────────────────────────┤
  │  Step 3: 分析原因                     │
  │  - 成功的关键因素是什么？              │
  │  - 失败的根因是什么？                  │
  │  - 哪些是可控的？哪些不可控？           │
  ├─────────────────────────────────────┤
  │  Step 4: 总结规律                     │
  │  - 学到了什么？                        │
  │  - 下次怎么做更好？                    │
  │  - Action Items 是什么？              │
  └─────────────────────────────────────┘
```

### 1.2 5Why 根因分析

```
5Why 示例：项目延期 2 周

  Why 1: 为什么项目延期？
  → 有 3 个功能开发超时

  Why 2: 为什么开发超时？
  → 需求在开发中途变更了 2 次

  Why 3: 为什么需求变更？
  → 需求评审时没有覆盖边界条件

  Why 4: 为什么评审没覆盖边界条件？
  → 产品给的需求文档没有边界条件描述

  Why 5: 为什么需求文档缺少边界条件？
  → 没有需求文档模板和评审检查清单

  根因：缺乏需求文档模板和评审 Check List
  Action：制定需求文档模板 + 评审检查清单
  负责人：产品负责人
  截止日：下个迭代前

  5Why 的原则：
    - 逐层追问，不跳步
    - 追到「流程/机制」层面（不是追到某个人）
    - 根因应该是可改进的流程问题
    - 避免追到「人不靠谱」（那不是根因）
```

## 二、复盘会议组织

### 2.1 会议准备

```
复盘会议准备清单：

  时机：
    - 项目上线后 1 周内
    - 里程碑结束
    - 重大事故后 3 天内
    - 不要拖太久（记忆会模糊）

  参与人：
    - 核心团队（PM/RD/QA/设计）
    - 利益相关方（可选）
    - 控制在 8 人以内

  准备材料：
    - 项目时间线（关键节点 + 事件）
    - 数据指标（计划 vs 实际）
    - 问题和风险清单
    - 每人提前写 3 条「做得好」+ 3 条「可改进」

  环境设置：
    - 白板/便利贴（可视化讨论）
    - 2 小时（太短不够深入，太长疲劳）
    - 安全氛围（不追责，只改进）
```

### 2.2 会议流程

```
复盘会议流程（2 小时版）：

  00:00-00:10  开场：明确复盘目的（学习，不追责）
  00:10-00:20  回顾目标：当初的目标和预期
  00:20-00:40  评估结果：用数据说话（计划 vs 实际）
  00:40-01:20  分析原因：分组讨论（做得好 vs 可改进）
  01:20-01:40  总结规律：提炼可复用的经验
  01:40-02:00  Action Items：明确负责人和截止日

  关键原则：
    1. 对事不对人（讨论「为什么」不讨论「谁的错」）
    2. 用数据说话（不凭感觉）
    3. 人人发言（每个人都要参与）
    4. 聚焦改进（而不是追究责任）
    5. Action Items 必须可执行、可跟踪
```

## 三、Action Item 跟踪

```
Action Item 跟踪闭环：

  复盘会议产出 Action Items → 录入任务系统 → 追踪到完成

  Action Item 表格：

  #  改进项                         负责人  截止日    状态
  1  制定需求文档模板                 张三    06-20   ✅ 完成
  2  代码评审增加安全检查项            李四    06-22   ✅ 完成
  3  慢查询告警阈值降到 500ms         王五    06-25   🔄 进行中
  4  每周四技术分享会                 赵六    07-01   📅 待开始
  5  建立测试环境数据准备脚本          钱七    07-05   📅 待开始

  跟踪机制：
    - 每周站会同步 Action Item 进度
    - 下次复盘首先检查上次 Action Items
    - 未完成的需说明原因
    - 连续 2 次未完成 → 升级处理

  知识沉淀：
    - 复盘报告存入团队 Wiki
    - 经验教训整理成 Check List
    - 新人入职学习复盘报告
    - 形成「团队经验库」
```

## 四、面试要点

### Q：项目复盘怎么做？

GRAI 四步法：回顾目标 → 评估结果 → 分析原因 → 总结规律。关键是分析原因用 5Why 追到流程/机制层面的根因，而不是停在「某人犯了错」。复盘会议 2 小时，8 人以内，对事不对人。产出 Action Items 录入任务系统，下周复盘先检查上次 Action Items。

### Q：复盘怎么避免变成甩锅会？

1. 开场明确目的：学习改进，不追责 2. 对事不对人：讨论「为什么发生」而非「谁的错」 3. 用数据说话：有客观数据不用主观评价 4. 聚焦改进：每个问题都有对应的改进措施 5. 安全氛围：负责人以身作则先反思自己的问题。

---

## 五、总结

```
复盘 = GRAI + 5Why + Action Items + 知识沉淀

四步法：回顾目标 → 评估结果 → 分析原因 → 总结规律
根因分析：5Why 逐层追问到流程/机制层面
会议：2h、8人内、对事不对人、用数据说话
闭环：Action Items 录入任务系统 → 周会跟踪 → 下次复盘检查

核心：复盘不是追责，是学习。不是总结过去，是改进未来。
```
